# Can I use oropt for my part?

oropt is mechanically general — it is **not** tied to the elevator linkage that
serves as the reference example. But it makes specific assumptions about the
model and the objective. This is an honest checklist of what works out of the box
and what does **not** (yet). The assumptions are baked into the deck editor
(`oropt/deck.py`), the config schema (`oropt/config.py`) and the solver seam
(`oropt/runner.py`), not into any one part.

## Works out of the box

- **A TET4 solid design part.** The optimiser rewrites exactly one
  `/TETRA4/<design_part_id>` element block, deleting the lowest-energy elements
  each iteration (`Deck._find_elem_block` targets `/TETRA4/<id>`; element cards
  are parsed as `id + 4 node ids`). Set `model.design_part_id` to that part.
- **An OpenRadioss implicit nonlinear deck that emits `/ANIM/ELEM/ENER`.** The
  per-element internal-energy density in that animation card *is* the BESO
  sensitivity — no adjoint, no per-element material interpolation. Without it
  there is no ranking signal. The full nonlinear model (e.g. LAW36 plasticity +
  contact) is solved every iteration; there is no linear-elastic simplification.
- **A compliance / stiffness objective.** The design is driven to remove the
  least strain-energy material (`beso.sensitivity`: `energy`, `vonmises`, or a
  `blend` of the two). This is the only objective family.
- **Per-load-case stress + displacement constraints.** Each load case sets its
  own `sigma_allow` (peak von-Mises [MPa]) and any number of `disp_constraints`
  (`{node_id, d_allow}`), enforced on OpenRadioss's high-fidelity values each
  iteration. Either may be left blank to leave that quantity unconstrained.
- **Multiple load cases on one shared mesh.** Several deck pairs that share the
  design-part element ids and differ only in their load cards, combined into a
  weighted-sum compliance objective and worst-case feasibility (see the README
  "How multiple load cases work").
- **A mixed model around the design part.** Rigid bodies, contacts
  (`/SURF/PART/EXT` is regenerated by OpenRadioss from the surviving elements),
  BCs, and other parts are copied through the deck **verbatim** — only the design
  `/TETRA4` block is edited. Those surrounding parts may be any element type; the
  restriction is on the *design* part only.
- **The Dockerised MUMPS-implicit build** as an alternative to a native Windows
  OpenRadioss install (`docker.enabled: true`) — same pipeline, no Intel
  oneAPI/MPI.

## Does NOT work yet

- **Shell or hexahedral (brick) design parts.** The design-part deck editor only
  finds and rewrites `/TETRA4` blocks; a `/SHELL` or `/BRICK` design part is not
  parsed as the design domain. (Brick/tet geometry *is* read for neighbour
  keep-out decks in `read_solid_geometry`, but that is forbidden-space geometry,
  not the optimised part.) Re-mesh the design part as TET4.
- **Non-compliance objectives** — eigenfrequency / NVH, buckling, thermal,
  fatigue/durability, or a true stress-*minimisation* objective. Stress today is
  a feasibility *constraint* (`sigma_allow`), not an objective; there is no modal,
  buckling or thermal objective. These are flagged as research/future or no-go in
  `docs/roadmap_2026.md` (items A5, A8, A9) — they need solver output or an
  eigensolve oropt does not consume.
- **Solvers other than OpenRadioss.** The runner launches the OpenRadioss starter
  + engine (native or the Docker MUMPS image); there is no Abaqus / LS-Dyna /
  Nastran / OptiStruct backend. oropt *consumes* decks converted from LS-DYNA by
  the sibling `k_to_rad_converter`, but the in-the-loop solver is OpenRadioss.
- **Multi-material or density-based SIMP design.** Element removal is a hard 0/1
  omit-the-card operation; there is no per-element modulus, so multi-material and
  SIMP-style interpolation are architecturally blocked (full argument in
  `docs/simp_spike.md`).

## Quick self-check

You can use oropt today if **all** of these are true:

1. Your design part is a TET4 solid part in an OpenRadioss implicit deck.
2. The engine deck emits `/ANIM/ELEM/ENER` (per-element energy).
3. Your goal is a stiffer / lighter part (compliance), with stress and/or
   displacement kept under limits — not a frequency/buckling/thermal target.
4. You can name the design part id, the BC node-group id, and (optionally) the
   constrained node id(s) — see `configs/elevator_linkage.yaml`.

If any of 1–3 fails, see the "Does NOT work yet" list above and
`docs/roadmap_2026.md` for where that capability sits on the backlog.
